# 上下文管理设计

## 目标

在有限的 Token 预算内保留对当前任务最有价值的信息，并优先牺牲可恢复、可重新读取或时效性较低的信息。系统采用两层机制：

1. 第一层：工具调用结果落盘，用于轻量预防。
2. 第二层：全量摘要压缩，用于容量兜底。

每次请求模型前，按第一层、第二层的顺序执行。

## 信息优先级

| 信息类型 | 上下文策略 | 原始数据策略 |
| --- | --- | --- |
| System / Developer 指令 | 始终完整保留 | 保留 |
| 当前用户目标、约束与最新请求 | 始终完整保留 | 保留 |
| 最近三轮对话 | 完整保留 | 保留 |
| 已修改文件、执行结论和关键错误 | 保留结构化摘要与路径 | 保留 |
| 大型工具结果 | 保留预览与归档引用 | 完整落盘 |
| 更早的会话、工具请求和工具结果 | 由全量摘要替代 | 保留，可恢复 |

一轮对话定义为一次 `user message -> assistant response（包含工具调用）`。

## 总体流程

```text
收到新请求
  -> 构建候选上下文
  -> 第一层：工具结果落盘与消息瘦身
  -> 估算 Token
  -> 未达到最大上下文的 75%：请求模型
  -> 达到最大上下文的 75%：第二层全量摘要压缩 -> 重新估算 -> 请求模型
```

## 第一层：工具调用结果落盘

### 触发条件

使用两个可配置阈值：

- `single_tool_result_token_threshold`：单个工具结果超过该值时落盘，建议初值为 `2,000 tokens`。
- `message_tool_results_token_threshold`：同一 assistant 消息内全部工具结果合计超过该值时落盘，建议初值为 `6,000 tokens`。

阈值应根据目标模型的最大上下文长度动态配置，不能作为固定常量。

### 处理规则

当单个工具结果超阈值时：

1. 将完整原始结果写入会话归档目录。
2. 写入工具名、调用参数摘要、时间、内容哈希、字节数和 Token 估算等元数据。
3. 用“摘要 + 前后预览 + 文件路径 + 哈希”替代上下文中的完整内容。

当同一消息内工具结果总量超阈值时：

1. 优先保留最近一轮中最关键的结果。
2. 对其余结果按 Token 大小降序依次落盘。
3. 每次落盘后重新计算，达到消息预算后停止。
4. 若仍超限，缩短保留结果的预览，但不得移除归档引用。

最近一轮结果不是无限豁免：极大的单项结果仍必须落盘，只是可保留更多预览。

### 上下文引用格式

```text
[工具结果已归档]
工具：exec_command
摘要：测试失败，共 12 个失败用例；主要集中于 auth 模块。
预览：
- FAIL auth/login.test.ts
- TypeError: token is undefined
归档路径：.context/tool-results/20260722_103015_exec_8f3c.txt
内容哈希：sha256:...
如需完整细节，请读取该文件。
```

建议元数据结构：

```json
{
  "tool_call_id": "call_123",
  "tool_name": "exec_command",
  "arguments_summary": "rg -n \\\"TODO\\\" src",
  "created_at": "2026-07-22T10:30:15+08:00",
  "token_estimate": 18432,
  "content_sha256": "...",
  "storage_path": ".context/tool-results/...txt",
  "preview": "前若干行 + 尾若干行",
  "status": "archived"
}
```

## 第二层：全量摘要压缩

### 触发条件

第一层完成后，估算待发送的整体上下文。当满足以下条件时执行压缩：

```text
estimated_context_tokens >= max_context_tokens * 0.75
```

估算必须包含当前输入、预留模型输出和预留工具调用空间，而不仅是历史消息。

### 压缩范围

完整保留：

- System / Developer 指令；
- 当前用户请求；
- 最近三轮完整对话；
- 当前任务需要的活跃文件引用、最新变更和关键错误；
- 已落盘工具结果的索引。

压缩为工作记忆：

- 更早的用户与 assistant 对话；
- 工具调用请求和工具结果；
- 可用且允许参与压缩的模型内部工作信息。

### 摘要内容

摘要应当是可执行的工作记忆，至少覆盖：

- 任务目标和成功标准；
- 用户偏好、约束和已作决定；
- 已完成与未完成事项；
- 修改过的文件及关键变更；
- 已执行命令、结论、错误和排查结果；
- 重要工具结果的归档路径；
- 不确定点、风险和待确认事项；
- 下一步行动。

建议固定为以下 Markdown 结构：

```markdown
## 任务状态
## 用户目标与约束
## 已完成工作
## 当前文件与关键变更
## 工具调用与可恢复资料
## 已知问题与结论
## 待办与下一步
## 重要边界条件
```

### 摘要 Prompt

摘要调用必须明确禁止工具调用：

```text
你是上下文压缩器。将给出的历史会话压缩为供后续执行模型使用的“工作记忆”。

要求：
1. 不得调用任何工具、函数、文件系统、网络或外部资源。
2. 不要回答用户，也不要提出新方案；仅提炼已有会话事实。
3. 保留任务目标、约束、决策、已完成工作、未完成工作、文件路径、命令结论、错误、重要数据和归档引用。
4. 区分“已确认事实”“模型推断”“待验证事项”。
5. 不得编造任何不存在的文件、结果、结论或用户偏好。
6. 对可从归档文件恢复的长内容，仅保留简要结论、路径、哈希和读取条件。
7. 输出结构化 Markdown，控制在 {summary_token_budget} Token 以内。
```

### 压缩后的边界消息

摘要后追加以下消息，明确摘要不是完整原文：

```text
【上下文边界】
早期会话已被压缩，详细原始工具输出可能仅保存在归档路径中。
摘要中的文件路径、工具结果路径和哈希可用于恢复细节。
如当前任务需要精确代码、完整日志、原始响应或文件内容，必须重新读取对应文件或归档，不能根据摘要臆测。
```

## 原始信息保留策略

压缩只将原始信息从模型 prompt 中移除，不删除会话存储中的原始数据。

原始信息必须保留，因为摘要存在信息损失；调试、审计、重放、用户追问以及恢复精确参数或日志都依赖原始记录。

| 数据 | 数据库 / 磁盘 | 发送给模型 |
| --- | --- | --- |
| 原始消息 | 保留，带压缩代次和索引 | 压缩后不直接发送 |
| 原始工具输出 | 完整归档 | 仅预览和路径 |
| 压缩摘要 | 保留版本链 | 发送 |
| 最近三轮 | 保留 | 完整发送 |

可配置生命周期：热数据保留当前会话原始记录与工具归档；温数据长期保存摘要与索引；冷数据按用户配置、存储配额及合规要求清理。敏感内容需加密、隔离访问，并支持主动删除。

## Token 预算

```text
最大上下文预算
- System / Developer 指令
- 当前用户请求
- 预留模型输出
- 预留工具调用
= 可用历史上下文预算
```

Token 估算优先使用目标模型 tokenizer；无法精确计算时使用保守近似估算。

## 异常与降级

- 摘要调用失败：保留最近三轮，进一步扩大工具结果落盘范围，并生成最小结构化状态。
- 归档写入失败：不得丢弃原结果；改为截断预览并标记“未归档”，或阻止后续请求。
- 摘要后仍超限：进一步缩短旧摘要，不压缩 System / Developer 指令和最近三轮。
- 引用文件不存在：标记引用失效，禁止模型假定其内容仍可读取。
- 多次压缩：记录摘要版本、来源范围与归档索引，确保可追溯。

## 验收标准

1. 单个超大工具结果不会直接占满上下文。
2. 单条 assistant 消息内的工具结果总量受预算约束。
3. 上下文达到最大 Token 的 75% 时自动全量摘要。
4. 压缩后完整保留最近三轮对话。
5. 摘要模型被明确禁止调用工具。
6. 压缩后附加“需要细节时重新读取”的边界消息。
7. 原始消息和工具结果可通过数据库或归档文件恢复。
8. 后续模型可以根据摘要继续执行，并按路径恢复精确细节。
